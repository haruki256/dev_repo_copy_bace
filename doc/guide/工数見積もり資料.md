# 工数見積もり資料

## 不確実性フレームワーク

— 簡易・正確 2方式による工数予測モデル —

# 1. 概要

本ドキュメントは、ソフトウェア開発プロジェクトの工数見積もりにおける「不確実性」を定量的に評価するためのフレームワークである。

不確実性の算出に「簡易方式」と「正確方式」の2つを用意している。

- **簡易方式**：想定工数（人日）を入力するだけで、統計データに基づく不確実性係数を即座に算出できる。見積もりの初期段階で使う。
- **正確方式**：不確実性を「認知不確実性（Known Unknowns）」と「非認知不確実性（Unknown Unknowns）」に分離し、個別に評価することで、より精度の高い工数範囲を算出する。

## 1.1 見積もりワークフロー

本フレームワークを使った見積もりは、以下の手順で行う。

```text
Step 1：想定工数を出す
  機能ごとに想定時間を洗い出し、合算する。
  想定工数(人日) = 想定時間の合計 ÷ 1日あたりの開発可能時間

Step 2：実効人数を出す
  人数の妥当性チェック：想定工数 ÷ 人数 ≥ 5人日 を満たすこと。
  コミュニケーションパス = (人数 - 1) × 0.01
  コミュニケーションコスト = 0.98（最良）/ 0.95（普通）/ 0.90（最低）
  実効人数 = 人数 × (1 - コミュニケーションパス) × コミュニケーションコスト

Step 3：想定日数・期間係数・固定BFを出す
  想定日数 = 想定工数 ÷ 実効人数
  期間係数 = 1.2 + 想定日数 ÷ 8000
  固定BF  = 想定日数 ÷ 10

Step 4：最短を出す
  最短日数 = 想定日数 × 1.2（バッファ）
  最短工数 = 最短日数 × 人数
  ※ 最短は理論上のベストケース。計画のターゲットにはしないこと。

Step 5：最大を出す（簡易 or 正確を選択）
  【簡易方式】
    最大日数 = (期間係数a × 想定日数 × 1.2 + 固定BF) × 期間係数b
  【正確方式】
    認知不確実性・非認知不確実性をスコアリングする（→ 3.3節）
    最大日数 = (認知 × 非認知 × 想定日数 × 1.2 + 固定BF) × 期間係数b
  最大工数 = 最大日数 × 人数

Step 6：カレンダーに当てはめる
  最短日数・最大日数を営業日ベースでカレンダーに当てはめ、
  完了予定日の範囲を出す。
```

```text
Step 7：目安日数を出す
  目安日数 = (最短日数 + 最大日数) ÷ 2
  目安工数 = 目安日数 × 人数
```

ブレ幅が許容範囲であれば、目安日数をターゲットとしてプロジェクトを進める。ブレ幅が大きすぎる場合は、Step 5の前に戻って不確実性を下げるアクション（仕様の詰めなおし、技術検証、コミュニケーション改善等）を検討する。

## 1.2 不確実性とは

不確実性とは、見える化できていないため推測しきれていない部分のことである。見える化できていない状態は以下の2つに分類される。

| 分類 | 定義 |
| --- | --- |
| 認知不確実性 | 認知しているが推測しきれていない不確実性。見えているが、理解や解像度が足りていない部分。 |
| 非認知不確実性 | 認知していない不確実性。存在自体を認識できておらず、見えていない部分。 |

## 1.3 学術的背景

本フレームワークは、以下の2つの学術的体系を統合している。

Epistemic / Aleatory Uncertainty（認識論的 / 偶然的不確実性）

Epistemic不確実性は知識不足が原因であり、調査・学習により削減可能。Aleatory不確実性はプロセスに内在するランダム性であり、削減不可能。

Rumsfeld Matrix（Known Knowns / Known Unknowns / Unknown Unknowns）

Known Unknownsは確率的に見積もれる不確実性、Unknown Unknownsはプロジェクトマネージャーが認識すらしていない不確実性。後者はさらにKnowable（探せば見つけられた）とUnknowable（真に予見不可能）に分かれる。

| 本フレームワーク | 対応する概念 | 性質 |
| --- | --- | --- |
| 認知不確実性 | Known Unknowns + Epistemic | 調査・POC・設計で削減可能 |
| 非認知不確実性 | Unknown Unknowns + Aleatory | 削減しにくいが、潜みやすい場所は予測可能 |

# 2. 計算式

## 2.1 全体構造

```text
実効人数 = 人数 × (1 - コミュニケーションパス) × コミュニケーションコスト
想定日数 = 想定工数(人日) ÷ 実効人数
固定BF = 想定日数 ÷ 10
期間係数 = 1.2 + 想定日数 ÷ 8000

最短日数 = 想定日数 × バッファ(1.2)
最短工数 = 最短日数 × 人数
```

本フレームワークでは、期間係数を役割に応じて2つに区別する。

- **期間係数a**：不確実性の簡易代替。簡易方式で認知・非認知不確実性を個別評価しない代わりに使う。
- **期間係数b**：期間ごとの構造的ズレのベース。バッファや不確実性を見込んだ計画でも、期間が長いほどそこからさらにズレるという統計的事実を反映する。どちらの方式でも外側に掛かる。

計算式は同じ `1.2 + 想定日数 ÷ 8000` だが、果たす役割が異なる。

期間係数が不確実性の簡易代替として機能する理由：期間が長いプロジェクトほど不確実性が増大する傾向があるため、期間の長さがそのまま不確実性の代理指標になる。また、想定工数ではなく想定日数を入力に使うのは、人数が多ければ期間が短くなり、人数によって不確実性を「希釈」できるという実態を反映するためである。

統計データはプロジェクトの暦期間（日数）に基づくため、式全体を想定日数（= 想定工数 ÷ 実効人数）ベースで計算し、最後に人数を掛けて人日に変換する。

最大工数の算出には2つの方式がある。

```text
【簡易方式】
最大日数 = (期間係数a × 想定日数 × 1.2 + 固定BF) × 期間係数b
最大工数 = 最大日数 × 人数

  期間係数a = 不確実性の簡易代替（認知・非認知を個別評価しない代わり）
  期間係数b = 規模による構造的ズレ（どんな見積もりでも発生する）

【正確方式】
最大日数 = (認知不確実性 × 非認知不確実性 × 想定日数 × 1.2 + 固定BF) × 期間係数b
最大工数 = 最大日数 × 人数

  内側 = 認知×非認知（不確実性を個別評価済み）
  期間係数b = 規模による構造的ズレ
```

簡易方式は不確実性を個別評価しないため、期間係数aが不確実性の代替も兼ね、期間係数が2回掛かる。正確方式は認知×非認知が内部の不確実性を担当するため、期間係数bは外側の1回のみ。

メモ：不確実性が高ければ高くなるほど、最大工数も高くなるため、不確実性を下げることで、最大工数を抑えられる。

## 2.2 各項の定義

想定工数（人日）

機能ごとに何時間で終わるかの合算を、1日あたりの開発可能時間で割ったもの。

```text
想定工数(人日) = 想定時間の合計 / 1日あたりの開発可能時間

1日あたりの開発可能時間 = 業務時間 - 開発外時間
（例：会議・質問対応・運用対応・障害対応等を除く。目安：4時間程度）
```

❗ 一日あたりの開発可能時間がずれると全体が大きくずれる。組織の実態に合わせて正確に設定すること。

人数の妥当性チェック

```text
想定工数 ÷ 人数 ≥ 5人日 を満たすこと。
```

一人あたり5人日未満の仕事しかない場合、タスク分割のオーバーヘッドがメリットを上回る。満たさない場合は、人数を減らすか、タスクの粒度を見直すこと。

実効人数

人数が増えるとコミュニケーションによるロスが発生する。本フレームワークでは、コミュニケーションパス（定量）とコミュニケーションコスト（定性）の2つで補正する。

```text
実効人数 = 人数 × (1 - コミュニケーションパス) × コミュニケーションコスト
```

**コミュニケーションパス**

自分以外のメンバーとのやり取りによるロス。1人増えるごとに1%のロスが発生する。

```text
コミュニケーションパス = (人数 - 1) × 0.01
```

**コミュニケーションコスト**

チームの状態によるコミュニケーション効率。確認待ち、レビュー待ち、伝達ロス等を反映する。開発外時間（会議・質問対応等）は1日あたりの開発可能時間の定義で既に除外されており、認識齟齬による手戻り等は認知不確実性が吸収するため、ここでは純粋なコミュニケーション効率のみを評価する。

| チーム状態 | コスト | 目安 |
| --- | --- | --- |
| 最良チーム | 0.98 | 信頼関係が厚く、阿吽の呼吸で進められる |
| 普通 | 0.95 | デフォルト値。一般的なチーム |
| 最低チーム | 0.90 | 関係が悪い、情報共有が滞りがち |

**金額換算の重要性**

コミュニケーションコストは日数で見ると軽微に見えるが、「人数 × 単価 × 日数」で金額換算すると無視できない額になる。

例：250人日のプロジェクト、6人チーム、1人1日単価1.5万円の場合
- 補正なし：250 ÷ 6 = 41.7日
- 最低チーム（パス+コスト）：250 ÷ (6 × 0.95 × 0.90) = 48.7日
- 差分：+7日 → **1.5万 × 6人 × 7日 = 63万円のロス**
- 年間10案件なら630万円。積み重ねで会社を潰しかねない額になる。

PMとしては、以下を常に意識すること。
- 必要コミュニケーション人数を最小限にする（意思決定者・確認者を絞る、自律的に判断できる範囲を明確にする）
- チームの状態を良好に保つ（チームビルディング、ドキュメント整備、仕様の事前確定）

これらの施策は認知不確実性の削減とも直結しており、二重に効果がある。

想定日数

プロジェクトの暦期間（営業日）。想定工数を実効人数で割ったもの。式の基本単位であり、期間係数・固定BF・最大日数の算出に使う。

```text
想定日数 = 想定工数(人日) ÷ 実効人数
（例：300人日を実効人数1.94人で行う → 300/1.94 ≈ 155日）
```

バッファ（1.2）

工数に遊びを作るための時間。作業切り替え、集中力の波、細かな調整等、どうやっても発生するオーバーヘッドを吸収する。最短工数・最大工数ともに1.2で統一。

バッファは不確実性の吸収とは別の役割。不確実性は不確実性係数が担当する。

固定バッファ

有給休暇によるプロジェクト期間の延長。10営業日あたり1日を加算する。

```text
固定バッファ = 想定日数 / 10
（例：想定日数100日 → 100/10 = 10日加算）
```

# 3. 不確実性の算出

## 3.1 期間係数

期間係数は両方式で共通の計算式を用いるが、果たす役割が異なる。

- **期間係数a**：簡易方式で認知・非認知不確実性を個別評価しない代わりの簡易代替
- **期間係数b**：バッファや不確実性を見込んだ計画でも発生する、規模による構造的ズレ

```text
想定日数 = 想定工数(人日) ÷ 実効人数
期間係数 = 1.2 + 想定日数 ÷ 8000
```

覚え方：**「1.2に、想定日数を8000で割った値を足す」**

統計データはプロジェクトの暦期間（日数）に基づくため、人日ではなく想定日数（= 想定工数 ÷ 実効人数）を入力する。同じ人日でも、少人数で長期間になるほうが期間係数は大きくなる。これは、人数が多ければ期間が短くなり、不確実性を希釈できるという実態を反映している。

| 想定日数 | 期間目安    | 期間係数 | 統計データの遅延倍率 | 誤差    |
| -------- | ---------- | ---------- | --------------- | ----- |
| 20       | 1ヶ月        | 1.20       | 1.20            | 0     |
| 60       | 3ヶ月        | 1.21       | 1.21            | 0     |
| 120      | 6ヶ月        | 1.22       | 1.23            | -0.01 |
| 240      | 1年         | 1.23       | 1.24            | -0.01 |
| 480      | 2年         | 1.26       | 1.27            | -0.01 |
| 1,200    | 5年         | 1.35       | 1.38            | -0.03 |
| 2,400    | 10年        | 1.50       | 1.50            | 0     |

例：300人日のPJを行う場合（コミュニケーション補正は省略）
- 2人で行う → 想定日数 = 150日 → 期間係数 = 1.2 + 150/8000 = **1.219**
- 5人で行う → 想定日数 = 60日 → 期間係数 = 1.2 + 60/8000 = **1.208**
→ 同じ人日でも、期間が短いほうが期間係数は小さくなる。

## 3.2 簡易方式

想定工数（人日）と人数だけで最大工数を算出する方式。見積もりの初期段階や、詳細な分析が不要な場合に使う。

```text
最大日数 = (期間係数a × 想定日数 × 1.2 + 固定BF) × 期間係数b
最大工数 = 最大日数 × 人数
```

期間係数が2回掛かる理由：
- **期間係数a**：認知・非認知不確実性を個別評価しないため、その簡易代替として機能する。期間が長いほど不確実性が増大する傾向を利用している。
- **期間係数b**：バッファや不確実性を見込んだ計画でも発生する構造的ズレ

統計データの根拠（参照元テーブル）：

| 計画期間      | 計画中央値(日) | 平均遅延日数   | 遅延率    | 遅延倍率 | 参考件数   | 主要データソース                        |
| ----------- | ---------- | ---------- | ------- | ------ | -------- | ---------------------------------- |
| 0〜1か月      | 15日        | 約3日       | ~20%   | 1.20   | 数千件規模  | ScienceDirect 2024（30万アクティビティ）    |
| 1〜3か月      | 60日        | 約13日      | ~21%   | 1.21   | 数千件規模  | ScienceDirect 2024                  |
| 3〜6か月      | 135日       | 約30日      | ~22%   | 1.23   | 数百件規模  | ScienceDirect 2024 + Buildertrend   |
| 半年〜1年      | 274日       | 約66日      | ~24%   | 1.24   | 5,000件+ | Standish CHAOS + McKinsey           |
| 1〜2年        | 548日       | 約143日     | ~26%   | 1.27   | 2,500件+ | Standish CHAOS + Flyvbjerg          |
| 2〜5年        | 1,279日     | 約453日     | ~34%   | 1.38   | 800件+   | Flyvbjerg 16,000件DB                 |
| 5〜10年       | 2,740日     | 約1,233日   | ~45%   | 1.50   | 100件+   | Flyvbjerg（McKinsey 50%遅延）          |

## 3.3 正確方式

認知不確実性と非認知不確実性を個別に評価する方式。プロジェクトの特性を詳細に分析する場合に使う。

```text
最大日数 = (認知不確実性 × 非認知不確実性 × 想定日数 × 1.2 + 固定BF) × 期間係数b
最大工数 = 最大日数 × 人数
```

認知×非認知が内部の不確実性を担当し、期間係数bは外側の構造的ズレのみを担当する（期間係数は1回のみ）。

### 3.3.1 認知不確実性（1.0〜2.0）

「見えているが推測しきれていない」範囲の広さ。Known Unknownsに対応し、調査・POC・設計によって削減可能な不確実性。

判断軸

仕様の確定度：要件定義がどの程度完了しているか

技術の習熟度：チームが使用技術にどの程度精通しているか

要件の安定度：ステークホルダーの合意が安定しているか、変更が予想されるか

コミュニケーション環境：チーム内の情報共有がスムーズか、確認・意思決定の経路が整理されているか

スコアリング基準（総合判断）

| スコア | 基準                       |
| --- | ------------------------ |
| 1.0 | 仕様確定済み、技術選定済み、類似実績あり     |
| 1.3 | 大枠は決まっているが詳細仕様が一部未定      |
| 1.5 | 仕様の半分程度が未確定、または未経験技術を含む  |
| 1.8 | 要件自体が流動的、ステークホルダーの合意が不安定 |
| 2.0 | ほぼ白紙、R&D要素が大きい           |

### 3.3.2 非認知不確実性（1.0〜2.0）

「存在自体を認知していない」不確実性。Unknown Unknownsに対応し、削減しにくいが、構造的に潜みやすい場所は予測できる。直接測れないため、間接指標で判断する。

**重要：非認知不確実性はプロジェクト期間が長くなるほど自然に高くなる。** 期間が長いほど、認知していなかった問題が顕在化する確率が上がるためである。スコアリング時に期間の長さも考慮すること。

判断軸

新規性：類似プロジェクトの経験があるか。ないほどUnknown Unknownsが潜みやすい

組織横断の複雑さ：複数チーム・部署間の調整が必要か。コミュニケーション障壁がUnknown Unknownsの温床

外部依存：外部API、サードパーティ、法規制等、自分たちでコントロールできない要素の数と影響度

要素間の相互作用：コンポーネント間の予期しない干渉リスク。複雑なシステムほど高い

プロジェクト期間：期間が長いほど、想定外の要因（人員変動、技術陳腐化、市場変化等）が発生しやすい

スコアリング基準（総合判断）

| スコア | 基準                      |
| --- | ----------------------- |
| 1.0 | 枯れた技術、社内に知見豊富、外部依存なし、短期PJ    |
| 1.3 | 一部新規技術あるが、ドキュメントや事例が十分  |
| 1.5 | 外部API依存あり、または組織横断の調整が必要 |
| 1.8 | 複数の新規技術の組み合わせ、法規制の影響あり、長期PJ  |
| 2.0 | 前例なし、技術的に未踏領域、外部要因が多数   |

| 5年（1200人日・1人）  | 1,200   | 1    | 1,200    | 120    | 1.350   | 2,786    | **2,786** | 2.32倍 | 長期PJの安全見積もり | ◎  |
| 10年（2400人日・1人） | 2,400   | 1    | 2,400    | 240    | 1.500   | 6,840    | **6,840** | 2.85倍 | メガPJの構造的ズレ   | ◎  |

簡易方式では期間係数aが不確実性の代替も兼ねるため、2回掛かる。同じ人日でも人数が増えれば想定日数が短くなり、期間係数が小さくなる（120人日・3人の行を参照）。

※ 検証テーブルでは、コミュニケーション補正の影響を明確にするため1人チームで計算している。複数人チームの場合は実効人数を用いること。

## 4.2 正確方式の検証

式：`最大日数 = (認知 × 非認知 × 想定日数 × 1.2 + 固定BF) × 期間係数b`、`最大工数 = 最大日数 × 人数`

| ケース        | 想定工数 | 人数 | 想定日数 | 固定BF | 認知  | 非認知 | 期間係数 | 最大日数 | 最大工数     | 倍率   | 現実感           | 判定 |
| ------------ | ------- | ---- | -------- | ------ | ---- | ---- | ------- | -------- | ---------- | ----- | --------------- | -- |
| 2ヶ月・明確     | 40      | 1    | 40       | 4      | 1.1  | 1.1  | 1.205   | 73.5     | **73.5**   | 1.84倍 | 小規模の安全マージン | ◎  |
| 6ヶ月・普通     | 120     | 1    | 120      | 12     | 1.5  | 1.3  | 1.215   | 298.9    | **298.9**  | 2.49倍 | 半年PJの現実的最悪 | ◎  |
| 2年・不透明     | 480     | 1    | 480      | 48     | 1.5  | 1.5  | 1.260   | 1,357    | **1,357**  | 2.83倍 | 中規模の厳しめ見積もり | ◎  |
| 10年・典型     | 2,400   | 1    | 2,400    | 240    | 1.5  | 1.5  | 1.500   | 8,100    | **8,100**  | 3.38倍 | メガPJ 3倍超に到達 | ◎  |
| 10年・最悪     | 2,400   | 1    | 2,400    | 240    | 2.0  | 2.0  | 1.500   | 14,040   | **14,040** | 5.85倍 | メガPJの破綻ケース | ◎  |

正確方式ではプロジェクト固有のリスクを内部に反映し、その外側に期間係数b（構造的ズレ）が掛かる。10年の典型ケースで3.4倍、最悪で5.9倍。Flyvbjergの報告（ITプロジェクトで平均45%過延、最悪400%超）とも整合。

# 5. 適用例

ECサイトリニューアル（3ヶ月想定）

プロジェクト概要

既存ECサイトをNext.jsでフルリニューアル。決済は外部API（Stripe）に移行。デザインは7割確定、残り3割は進行中に決定。チームにNext.js経験者は1名、他はVue.js経験者。社内の在庫管理システムとのAPI連携あり（仕様書が古い）。

```text
想定工数 = 60人日（2人 × 3ヶ月、1ヶ月=20人日）
コミュニケーションパス = (2 - 1) × 0.01 = 0.01
コミュニケーションコスト = 0.95（デフォルト）
実効人数 = 2 × (1 - 0.01) × 0.95 = 2 × 0.99 × 0.95 = 1.881
想定日数 = 60 ÷ 1.881 ≈ 31.9日
期間係数 = 1.2 + 31.9 ÷ 8000 = 1.204
固定BF = 31.9 ÷ 10 ≈ 3.2日
```

## 5.1 簡易方式での算出

```text
最大日数 = (1.204 × 31.9 × 1.2 + 3.2) × 1.204
        = (46.1 + 3.2) × 1.204
        = 49.3 × 1.204
        ≈ 59.4日
最大工数 = 59.4 × 2 = 118.8人日

最短日数 = 31.9 × 1.2 = 38.3日
最短工数 = 38.3 × 2 = 76.6人日
```

結論（簡易）：38日〜59日（工数 77人日〜119人日）の範囲。

## 5.2 正確方式での算出

スコアリング

| 項目       | スコア  | 判断根拠                                         |
| --------- | ---- | -------------------------------------------- |
| 認知不確実性  | 1.5  | 仕様3割未定 + Next.js未経験者多数 + デザイン流動的             |
| 非認知不確実性 | 1.4  | ECリニューアル実績あり + Stripeはドキュメント豊富 + 在庫API仕様書が古い（3ヶ月の短期PJのため期間リスクは低い） |

```text
最大日数 = (1.5 × 1.4 × 31.9 × 1.2 + 3.2) × 1.204
        = (80.4 + 3.2) × 1.204
        = 83.6 × 1.204
        ≈ 100.6日
最大工数 = 100.6 × 2 = 201.2人日

最短日数 = 31.9 × 1.2 = 38.3日
最短工数 = 38.3 × 2 = 76.6人日
```

結論（正確）：38日〜101日（工数 77人日〜201人日）の範囲で提示する。

## 5.3 方式の比較

| 方式 | 最大日数 | 最大工数 | 想定工数の何倍か | 用途                              |
| ---- | ------- | ------- | ------------- | --------------------------------- |
| 簡易 | 59日    | 119人日  | 1.98倍        | 初期段階の概算、統計的な平均ケース      |
| 正確 | 101日   | 201人日  | 3.35倍        | リスク分析後の現実的な最悪ケース        |

正確方式のブレ幅が大きい場合は、不確実性を下げるために以下のような措置を検討する。これらの措置はコミュニケーションコストの削減にも直結し、二重の効果がある。

- 仕様の詰めなおし → 認知不確実性を下げる → 確認のやり取りが減りコミュニケーションパスも実質削減
- 技術の変更（習熟済みの技術に切り替え）→ 認知不確実性を下げる → 「これどうやるんですか」が減りコミュニケーションコストが改善
- POC・技術検証の実施 → 非認知不確実性を下げる
- 外部依存の削減 → 非認知不確実性を下げる
- 意思決定者・確認者を絞る → コミュニケーションパスを削減
- チームビルディング・ドキュメント整備 → コミュニケーションコストを改善


# 6. 実績フィードバック

## 6.1 目的

本フレームワークの期間係数は統計データから導出した汎用的な値（ベース値1.2）を使用している。実際のプロジェクト実績を記録し、ベース値をチームの実態に合わせてチューニングすることで、見積もり精度を継続的に向上させる。

## 6.2 チューニング対象

```text
期間係数 = ベース値 + 想定日数 ÷ 8000
              ↑
          ここだけ変える（初期値：1.2）
```

「想定日数 ÷ 8000」の部分は統計的な構造ズレであり、変更しない。チューニングするのは**ベース値のみ**。

- 実績が目安日数より短い傾向 → ベース値を下げる（例：1.2 → 1.15）
- 実績が目安日数より長い傾向 → ベース値を上げる（例：1.2 → 1.25）

## 6.3 記録テンプレート

案件完了時に以下を記録する。

```text
案件名：
想定工数（人日）：
人数：
使用方式：簡易 / 正確
認知不確実性（正確方式の場合）：
非認知不確実性（正確方式の場合）：
コミュニケーションコスト：
最短日数：
最大日数：
目安日数：
実績日数：
実績工数（人日）：
実績 ÷ 想定工数：    倍
着地位置：最短寄り / 目安付近 / 最大寄り / 最大超え
振り返りメモ：
```

## 6.4 活用方法

5〜10件の実績が溜まったら、以下を確認する。

**ベース値の調整判断**

実績が目安日数に対してどこに着地しているかの傾向を見る。

- 多くの案件が最短寄りに着地 → ベース値を下げる余地あり
- 多くの案件が最大寄り・最大超えに着地 → ベース値を上げるべき
- 目安付近にばらつく → 現在のベース値が適正

**チームの成長の可視化**

ベース値の推移はチームの改善度合いを示す。「去年はベース値1.25だったが、今年は1.15に下がった」のように、見積もり精度の向上やチーム力の成長を数字で示すことができる。


# 7. 用語集

| 用語            | 定義                                       |
| ------------- | ---------------------------------------- |
| 1人日           | 1日あたりの開発可能時間。業務時間から会議・質問対応・運用対応等を除いた時間   |
| 想定工数          | 機能ごとの想定時間の合算を1人日で割ったもの                   |
| バッファ          | 作業切り替え等のオーバーヘッドを吸収する遊び時間。不確実性の吸収とは別の役割   |
| 固定バッファ        | 有給休暇による工数減少。想定日数の1/10を加算（例：想定日数100日→10日加算）                  |
| 期間係数          | 統計データから導出した係数（1.2 + 想定日数/8000）。期間係数a（不確実性の簡易代替）と期間係数b（規模による構造的ズレ）の2つの役割がある。簡易方式では2回、正確方式では1回掛かる |
| 期間係数a         | 簡易方式で認知・非認知不確実性を個別評価しない代わりの簡易代替。期間が長いほど不確実性が増大する傾向を利用 |
| 期間係数b         | バッファや不確実性を見込んだ計画でも発生する、規模による構造的ズレのベース |
| 想定日数          | 想定工数(人日) ÷ 実効人数。式の基本単位。期間係数・固定BFの入力に使う       |
| 実効人数          | 人数にコミュニケーション補正を適用した値。人数 × (1 - コミュニケーションパス) × コミュニケーションコスト |
| コミュニケーションパス | 自分以外のメンバーとのやり取りによるロス。(人数 - 1) × 0.01 |
| コミュニケーションコスト | チームの状態によるコミュニケーション効率。最良0.98、普通0.95（デフォルト）、最低0.90 |
| 認知不確実性        | Known Unknowns。見えているが推測しきれていない不確実性。削減可能  |
| 非認知不確実性       | Unknown Unknowns。存在自体を認知していない不確実性。削減しにくい。期間が長いほど高くなる傾向 |
| Epistemic不確実性 | 知識不足が原因の不確実性。学習・調査で削減可能                  |
| Aleatory不確実性  | プロセスに内在するランダム性。削減不可能                     |
