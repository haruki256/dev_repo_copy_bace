# AI活用メソッド

## この資料の目的

自社の開発業務で AI を効果的に使うための運用方法をまとめる。Claude Code と GitHub Copilot の2つを使い分ける前提で、コンテキスト管理・設計作業・コードレビュー・実装補助の観点から整理する。

## 鉄則

### 1. コンテキストは資源

AIが「今見えている情報」には上限がある。載せれば全部読むが、載せすぎると精度が落ちる。必要な情報を厳選して渡し、重要なことは明示的に指示する。AGENTS.md に要点をまとめているのはこの原則に基づく。

セッションが長くなるほど文脈の質が劣化する「**コンテキストエントロピー**」という現象がある。AIが古い情報と新しい情報を混同し、存在しない関数を参照したり、変更済みのアーキテクチャに基づいた提案を始める。対策は「全部覚える」のではなく「何をどこに覚えたかだけ覚える」こと。

### 2. ai-stash はヒント、コードが事実

AIの記憶（ai-stash やセッション内の過去の発言）は時間とともに古くなる。先週正しかった情報が今週は間違っている可能性がある。AIが「こうなっているはず」と言っても、必ず実コードで検証してから行動する。

なお、モデルが新しい＝品質が高い、とは限らない。Anthropic社内の指標では、新モデルが旧モデルより虚偽主張率が悪化したケースがある（v8: 29-30% vs v4: 16.7%）。AIの出力を信用しすぎない理由の一つ。

### 3. 判断は人間がする

AIは選択肢の整理・調査・整合性チェックに強い。しかし「どの方法を採用するか」「このリスクを許容するか」といった判断は人間がする。AIはどこまでいってもアシスタントであり、意思決定者ではない。

## コンテキスト管理：「1箇所に書いて、各AIから参照する」

### 背景：AGENTS.md が共通規格になった

2026年現在、AI エージェント向けのコンテキストファイルは **AGENTS.md** が事実上の共通規格となっている（Linux Foundation 傘下の Agentic AI Foundation が管理、60,000以上の OSS プロジェクトで採用）。

ただし、**AGENTS.md の自動読み込み対応状況はツールによって異なる**。「共通規格＝全ツールが自動で読む」ではない点に注意が必要。

### 自動読み込みの実態（2026年4月時点）

各ツールには「自動でコンテキストに注入されるファイル」がそれぞれ決まっている。自動注入されるファイルの内容は毎ターン（APIリクエストごとに）コンテキストに含まれるため、確実に参照される。一方、自動注入されないファイルは AI が能動的に読みに行く必要があり、読まれない可能性がある。

| ツール | 自動読み込みファイル | AGENTS.md 自動読み込み | 備考 |
|---|---|---|---|
| **Claude Code** | CLAUDE.md | **未対応** | Issue #6235 で要望中（3,200+アップボート）。CLAUDE.md に「AGENTS.md を読め」と書いても読まれない場合がある（モデルの判断依存） |
| **GitHub Copilot** | `.github/copilot-instructions.md` | **条件付き** | Coding Agent は対応済。VS Code Chat は `chat.useAgentsMdFile: true` の設定が必要（デフォルト off）。Code Review は未対応 |
| **Gemini CLI** | GEMINI.md | **条件付き** | デフォルトは GEMINI.md のみ。`settings.json` の `context.fileName` に明示指定が必要 |
| **Cursor** | `.cursorrules` / `.cursor/rules` | **完全対応** | 設定不要。サブディレクトリのネストもサポート |
| **OpenAI Codex** | `codex.md` / AGENTS.md | **完全対応** | AGENTS.md の生みの親。階層的な読み込みをサポート |

**重要な教訓：** 「AGENTS.md を読め」というポインタだけをツール固有ファイルに書く運用では、ポインタを読んでも AGENTS.md を実際に開かないケースが発生する。これはバグではなく仕様（AIが能動的に読むかどうかはモデルの判断に依存する）。

### コンテキストファイルの運用方針

上記の実態を踏まえ、以下の方針を採用する。

**現在の方針：AGENTS.md の内容を各ツールの自動読み込みファイルに丸ごと入れる（各ファイルが自己完結）。**

AGENTS.md の自動読み込みが主要ツールで未対応または条件付きである以上、「AGENTS.md を読め」というポインタ方式では指示が読まれないリスクがある。確実性を最優先し、各ツールの自動読み込みファイルに指示を直接記載する。

```
現在の方針（全ファイル自己完結）:
AGENTS.md              — 共通規格としての正本（Cursor, Codex 等の対応ツール向け）
CLAUDE.md              — AGENTS.md と同じ内容を記載（Claude Code 向け）
GEMINI.md              — AGENTS.md と同じ内容を記載（Gemini CLI 向け）
copilot-instructions.md — AGENTS.md と同じ内容を記載（GitHub Copilot 向け）

将来の方針（各ツールが AGENTS.md を自動読み込みに対応した場合）:
AGENTS.md              — すべてここに集約
CLAUDE.md              — 「AGENTS.md を読め」ポインタのみ
GEMINI.md              — 同上
copilot-instructions.md — 同上
```

**将来のポインタ方式への移行条件：** 各ツールが AGENTS.md の自動読み込みに正式対応した時点で、そのツールの固有ファイルをポインタ方式に切り替える。移行はツール単位で段階的に行う。

### 各ファイルの役割分担

| ファイル | 置き場所 | 自動読み込み | 現在の運用 |
|---|---|---|---|
| **AGENTS.md** | プロジェクトルート | Cursor, Codex は自動。他は条件付きまたは未対応 | 共通規格としての正本。内容の原本はここで管理する |
| **CLAUDE.md** | プロジェクトルート | Claude Code（毎ターン自動注入） | AGENTS.md の内容を丸ごと記載 |
| **GEMINI.md** | プロジェクトルート | Gemini CLI（毎ターン自動注入） | 同上 |
| **copilot-instructions.md** | .github/ | GitHub Copilot | 同上 |

### 自動読み込みファイルの設計指針

自動読み込みファイル（CLAUDE.md 等）は毎ターンのコンテキストに含まれるため、以下を意識する：

- **小さく保つ**（大きいと毎ターンのコストが膨らむ）
- **「何を知っているか」ではなく「何をどこで調べるか」を書く地図として設計する**
- **行動規範**（コーディング規約、回答スタイル、禁止事項）を書いておくとAIの自律作業時の品質担保になる
- チーム全員のAIが同じ前提・同じ行動規範で動く。毎セッション冒頭で説明し直す手間が消える

### 現在の構成（本プロジェクト）

```
xx/
├── AGENTS.md                       ← 正本（内容の原本管理先）
├── CLAUDE.md                       ← Claude Code 用（AGENTS.md と同内容）
├── GEMINI.md                       ← Gemini CLI 用（AGENTS.md と同内容）
├── .github/copilot-instructions.md ← Copilot 用（AGENTS.md と同内容）
└── doc/AI/
    └── AI活用メソッド.md              ← この資料
```

### 更新ルール

- 指示内容を変更する場合 → **AGENTS.md（正本）を更新し、各ツールの自動読み込みファイルにも同じ変更を反映する**
- 新しい AI ツールを追加する場合 → そのツールの自動読み込みファイルを作成し、AGENTS.md の内容を丸ごと記載する
- あるツールが AGENTS.md の自動読み込みに対応した場合 → そのツールの固有ファイルをポインタ方式に切り替える

## AI への指示の出し方

### シンプルなアプローチを最初に試す

複雑な最適解を一発で求めず、まず動くシンプルな解を出して反復する方が品質が高い。

### 停止条件を明示する

曖昧な指示はAIがループに陥る原因になる。「何をもって完了とするか」を明示する。

```
# 悪い例：停止条件なし
このコードを改善して

# 良い例：完了条件を明示
src/payment/checkout.java のエラーハンドリングを修正して。
完了条件：全テストがパスし、NullPointerExceptionの可能性がないこと。
```

### コンテキストを汚染しない指示の出し方

AIのコンテキストウィンドウには容量制限がある。大量データを一括で読み込ませると、重要な情報が追い出される（キャッシュ汚染）。

```
# 悪い例：コンテキスト汚染
このログファイル全文を読んで問題を探して

# 良い例：必要な部分だけ読み込ませる
src/logs/payment.log の直近200行を読んで、
エラーレベルのエントリだけ抽出して原因を分析して
```

```
# 悪い例：ワーキングセット肥大
このプロジェクト全体をレビューして改善点を挙げて

# 良い例：対象を絞る
src/usecases/payment/ 配下の3ファイルをレビューして、
エラーハンドリングの一貫性を確認して
```

```
# 悪い例：大きな結果をコンテキストに残す
全テストの実行結果を見せて

# 良い例：結果をファイルに退避して要約だけ受け取る
全テストを実行して、結果をtest-results.mdに保存して。
失敗したテストの名前と失敗理由だけをここに報告して。
```

### ファイル編集はまず読んでから

ファイル編集は文字列マッチによる差分置換で行われる。「まずファイルを読んでから編集して」と指示するとマッチ失敗が減る。

### ファイル読み取りには暗黙の上限がある

ファイル読み取りは2,000行で打ち切られ、AI自身にはその警告が出ない。大きなファイルを扱うときは範囲を指定すること。

### タスク種類ごとの先読み指示

作業開始時に「何を先に読むか」を指示すると効率が上がる。

```
# 例：バグ修正時
まず関連するテストファイルと、エラーが出ているソースファイルを読んでから修正に取りかかって

# 例：機能追加時
まず doc/design/architecture.md と既存の類似機能の実装を読んでから設計して
```

## AIと協働するためのコード設計

AIが読み書きする前提でのコード設計の変化。

### 判断理由をコメントに書く

人間には自明でもAIには書かれていなければわからない情報がある。「なぜこの実装にしたか」の判断理由をコメントに残すと、AIが不要なリファクタを提案しにくくなる。

```java
// パフォーマンス要件により一括INSERTを使用（バッチサイズ100件はDB負荷テスト済み）
jdbcTemplate.batchUpdate(sql, batchArgs);
```

### ドキュメントの自動更新

コード変更のたびにAIにドキュメント更新を依頼する運用。ドキュメントとコードの乖離を防ぐ。AGENTS.md にドキュメント更新ルールを記載しておけば、AIが自動で対応する。

## 設計作業での AI 活用

### 複雑な設計の整理に使う

**効果があったこと：**
- **方法比較の壁打ち**：複数の設計方法のメリット・デメリットを整理し、見落としているリスクがないか確認する。人間が「こうだと思う」をぶつけて、AI が「この場合はどうなるか」と返すやりとりで設計の穴が見つかる
- **フロー図やシーケンス図の生成**：mermaid 形式でフロー図を生成し、認識合わせに使う。手で書くより速く、修正も簡単
- **資料の整合性チェック**：大きな設計資料の用語統一、条件の非整合、重複の検出。人間の目では見落としがちな不整合を機械的にチェックできる
- **外部レビューとの突き合わせ**：GPT でレビューした結果を Claude で評価し、妥当な指摘と過剰な指摘を判別する

**注意すべきこと：**
- **AI の出力を鵜呑みにしない**：特に既存コードとの整合性は必ず実コードで確認する。今回も IS_FIX vs SEND_GIFT_STATUS の乖離は AI が自力で気づいたのではなく、人間が「本当にこれで合ってる？」と疑って検証した結果見つかった、そもそも、複雑になればなるほど、AI が完全に正しい前提で設計を進めるのは危険。AI はあくまで「こうなっているはず」というヒントを出す役割。実際に調査先のソースを見たり、自分の違和感を大事にしたり、人間が主体的に関与して設計を進めることが重要。
- **用語はこちらから正す**：AI は資料に書いてある用語をそのまま使うので、元の資料が間違っていると間違ったまま進む（rk = atss の件）。ドメイン知識は人間が持っている前提で、違和感があったら指摘する
- **AI にすべてを任せない**：設計の判断は人間がする。AI は選択肢の整理と検証の補助。「方法③がいいと思う」という判断は人間がして、AI は「その場合このリスクがある」「このパスが抜けている」を返す役割

---

### 運用ルール

1. **ai-stash は「ヒント」であり「事実」ではない**：ai-stash に書いてあることと実際のコードが矛盾したら、コードが正
2. **定期棚卸し**：スプリントの区切りで ai-stash の内容を見直し、古い情報を削除・統合する

## 複数 AI 利用時の注意点

### 同じ作業を別の AI で二重にやらない

Claude Code で設計を進めている最中に、Copilot で同じファイルを別の方向に修正すると混乱する。役割を明確に分ける。

### AI 間でのコンテキスト共有はファイル経由

Claude Code の会話内容を Copilot に直接渡すことはできない。共有したい情報はファイル（設計資料、AGENTS.md、コミットメッセージ等）に書き出す。ファイルに書いてあれば、どの AI からでも参照できる。

### AI の出力を別の AI でレビューする

GPT でレビューした結果を Claude で評価した。異なる AI を使うことで、1つの AI のバイアスや見落としを検出できる。ただし、最終判断は人間がする。

## まとめ

1. **コンテキストは資源。AGENTS.md に1箇所管理**。ツール固有ファイルはポインタのみ。二重管理を防ぐ
2. **AI は設計の壁打ち相手・調査補助・整合性チェッカー**。判断は人間がする
3. **ai-stash はヒント、コードが事実**。AI の記憶を鵜呑みにせず、必ず実コードで検証する
4. **ドメイン知識は人間が持つ**。AI が使う用語や前提が間違っていたら正す
5. **ファイルに書けば共有できる**。AI 間のコンテキスト共有はファイル経由で行う
6. **シンプルに始めて反復する**。複雑な最適解を一発で求めない
