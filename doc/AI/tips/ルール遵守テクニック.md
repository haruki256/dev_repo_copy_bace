# LLM ルール遵守テクニック — AIエージェント非依存の汎用手法

## 概要

CLAUDE.md / AGENTS.md 等のルールファイルに記載した指示が、会話の進行に伴い無視される問題への対策をまとめる。
本ドキュメントは**特定のAIエージェントに依存しない汎用テクニック**に限定している。

## 背景

- LLM はプロンプトの指示数が増えるほど追従率が一様に劣化する
- 特にトーン・スタイル指示はユーザーの会話トーンに引きずられやすい（ミラーリング傾向）
- コンテキストが膨張するほど、ルールファイルの指示は薄れる（Context Rot）

---

## 1. 指示数の削減（最重要）

LLM が確実に追従できる指示数には上限がある。指示が多すぎると全体の追従率が一様に低下する。

**目安**
- 理想: 60行以下
- 上限: 200行
- 500行超: 大半が無視される

**削減の判断基準**
- 「この行がなければ AI が間違えるか？」— No なら削除
- リンター・フォーマッターで対応できるルールは記載不要（決定論的ツールの方が確実）
- AI が標準的に持つ知識と重複する内容は記載不要
- タスク固有の詳細は別ファイルに分離し、ポインタのみ記載（Progressive Disclosure）

**注意**: エージェントの性能差がある。高性能モデルでは不要でも、低性能モデルでは必要な指示がある。
削減は「どのエージェントでも明らかに不要」と言えるものに限定する。

**根拠**
- ETH Zurich の研究: 人間がキュレーションしたルールファイルは約4ポイントの成功率向上。LLM 自動生成のものは約3%低下
- HumanLayer の実例: 60行以下の CLAUDE.md で十分機能
- Codex CLI: プロジェクトドキュメントを 32KiB で切り捨てる仕様

---

## 2. 重要ルールの配置位置（Lost in the Middle 対策）

LLM はプロンプトの**先頭と末尾**の情報に最も注意を払い、**中間部の情報は30%以上精度が低下**する。

**対策: サンドイッチ配置**

```
[ファイル先頭] 最重要ルール（セキュリティ、トーン指示等）
[中間]         コーディング規約、参照ポインタ、補足情報
[ファイル末尾] 最重要ルールのリマインド
```

**根拠**
- Liu et al. (2023) "Lost in the Middle": 20文書QAタスクで、関連文書を位置5〜15に配置すると位置1/20と比較して30%以上の精度低下

---

## 3. 肯定形で記述する

LLM は否定指示に対して一貫してパフォーマンスが低い。スケールアップするほど悪化する。

**メカニズム**: トークン生成は「次のトークンを選択する」正の選択プロセス。否定プロンプトはトークンの確率をわずかに抑制するだけだが、肯定プロンプトは望ましいトークンの確率を積極的に増幅する。

| 否定形（効果低） | 肯定形（効果高） |
|---|---|
| カジュアルな表現を避ける | 敬体（です・ます調）で記述する |
| 過剰な横文字を避ける | 技術用語は日本語を優先する |
| Don't add features beyond what was asked | A bug fix stays scoped to the bug |
| Don't add docstrings to code you didn't change | Keep docstrings unchanged in code you didn't modify |

**最も効果的なパターン**: 肯定形で望ましい行動を指示し、補足として否定形のガードレールを添える併用型。

---

## 4. 事実形式で記述する

「〜してください」（指示形式）より「〜を使用する」（事実形式）の方が無視されにくい。

| 指示形式（弱い） | 事実形式（強い） |
|---|---|
| コードの品質に気をつけてください | ESLint + Prettier で品質を担保する |
| 丁寧な言葉遣いで回答してください | 回答は敬体（です・ます調）で記述する |
| テストを実行してください | コード修正後は Lint & Test を実行する |

---

## 5. MUST / NEVER / IMPORTANT の限定使用

すべてを強調すると何も強調されないのと同じになる。

**原則**: MUST / NEVER / IMPORTANT は**本当に重要なルール1〜3個**に限定する。

**効果を高める条件**: 理由（Why）を付記する。
```
IMPORTANT: 回答は敬体（です・ます調）で記述する。ユーザーのトーンに関わらずこのスタイルを維持する。
```
理由があると AI が一般化でき、明示されていないエッジケースでも正しく判断できる。

---

## 6. 具体的・定量的に記述する

曖昧な指示ほどコンテキスト増加時に無視されやすい。Yes/No で判定可能な基準にする。

| 曖昧（弱い） | 具体的（強い） |
|---|---|
| カジュアルな表現を避ける | 敬体（です・ます調）を使用する。体言止めは見出しのみ |
| コードをきれいに書く | 関数は50行以内。超える場合は分割する |
| テストを書く | 新規関数には単体テストを追加する |

---

## 7. If X, then Y 形式（条件付き指示）

条件付き指示は最も効果的な指示パターン。トリガーと行動を明示的に結合する。

```
- Python ファイル編集時 → ruff を実行する
- テーブル追加・カラム変更時 → DB_DESIGN.md を更新する
- ファイル/フォルダの追加・削除・移動時 → PACKAGE_STRUCTURE.md を更新する
```

---

## 8. 構造化（Markdown / XMLタグ）

長い文章の塊は読み飛ばされる。箇条書き・テーブルで構造化する。

**フォーマット選択**
- **Markdown**: 全エージェントで安全な選択（AGENTS.md 仕様自体が Markdown 前提）
- **XMLタグ**: 情報の混合・汚染を防ぐ明確な境界を作る。品質最大40%向上との報告あり
- **JSON**: GPT-3.5-turbo が好む傾向あるが、可読性が低く汎用性で劣る

汎用ルールファイルでは **Markdown を基本とし、特に重要な境界にのみ XMLタグを補助的に使う** のが現実的。

---

## 9. トーン指示が無視される問題への特化対策

トーン・スタイル指示は特に無視されやすい。ユーザーの会話トーンをミラーリングする傾向があるため。

**対策の組み合わせ**
1. ファイル冒頭にトーン指示を配置（Lost in the Middle 対策）
2. 事実形式で断言（「敬体で記述する」）
3. 「ユーザーのトーンに関わらず」を明記（ミラーリング抑制）
4. ファイル末尾でリマインド（サンドイッチ配置）
5. 具体例を示す（Few-shot 効果）

---

## 10. Progressive Disclosure（参照型分離）

全情報をルールファイルに詰め込まず、詳細は別ファイルへポインタで参照させる。

```markdown
## 参照先
- コーディング規約の詳細 → `doc/rules/CODING_RULES.md`
- アーキテクチャガイド → `doc/guide/ARCHITECTURE_GUIDE.md`
- ai-stash 書き出し手順 → `ai-instructions/stash-guide.md`
```

これはエージェント非依存。どのツールでも「指定されたファイルを読む」能力は持つ。
