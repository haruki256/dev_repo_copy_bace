# AIモデルのコンテキスト限界と指示追従性の劣化

本ドキュメントでは、LLM（大規模言語モデル）そのものが抱えるコンテキスト処理の限界（Lost in the middle現象）と、複数指示への追従性の低下、およびそれがエージェントと組み合わさった際に引き起こす相互作用（シナジーと悪循環）について解説します。

（参考記事: [AIエージェントにおけるコンテキストエントロピー](コンテキストエントロピー.md)）

## 1. LLMにおけるコンテキストの真実と限界

昨今の言語モデル（Gemini 1.5 Pro, Claude 3.5 Sonnet, GPT-4oなど）は、数十万〜数百万トークンという巨大な「コンテキストウィンドウ」を持っていますが、**「入力をすべて同等に理解し、記憶している」わけではありません。**

### Lost in the Middle（中間情報の喪失）現象
研究（スタンフォード大等の論文など）により、LLMは**「コンテキストの最初（Primacy effect）と最後（Recency effect）の情報は適切に処理できるが、中間（Middle）にある情報は無視したり検索に失敗したりする」**という「U字型」のパフォーマンス曲線を描くことが分かっています。

コンテキストウィンドウが広いからといって大量のファイルやチャット履歴（エントロピーの高い情報）を送ると、中間に埋もれた重要な仕様やルールがモデルに見落とされるリスクが急増します。

### 指示追従性の低下（Curse of Instructions）
タスクが複雑になり、処理すべき制約（コーディング規約、特定のフォーマット指定、禁止事項など）が増えるほど、モデルが「すべての指示」を同時に満たす確率は指数関数的に下落します。
長いコンテキストの中に散りばめられた細かい指示は薄まりやすく、結果として「動作するが、ルールを逸脱したコード」が生成されやすくなります。

## 2. 自然言語によるコンテキスト圧縮（要約）の実態と限界

長い履歴を処理するため、エージェント側で「蓄積したログを要約（圧縮）して次のプロンプトに渡す」というアプローチが取られます。これは「自然言語でのプロンプト圧縮（Semantic Summarization）」と呼ばれます。

### 各エージェントの圧縮（要約）アプローチの違い

実際に主要なエージェントは、増大する履歴に対して以下のような異なるアプローチで対処しています：

1. **自動要約・コンパクション型（Claude Codeなど）**
   - **仕組み**: コンテキストが限界（例えば200k付近）に近づくと、自動的に「Compaction（圧縮）」プロセスが走ります。過去の数万トークンに及ぶ会話履歴やツールの実行結果を再度LLMに読み込ませ、「現在のタスク」「未解決のバグ」「重要な設計判断」といった数千トークンの箇条書きや要約文に変換します（手動の `/compact` コマンド等も存在します）。
   - **メリット**: 古いログのノイズが消え、S/N比が劇的に回復します。
2. **無圧縮・全保持型（Gemini CLIなど）**
   - **仕組み**: モデルのもつ1M〜2Mの超広大なコンテキストウィンドウを活かし、要約などの加工を行わず、文字通り「最初の会話からの全履歴」をそのまま後ろのプロンプトに繋げて送り続けます。
3. **スライディングウィンドウ・忘却型（Codex系CLI / エディタ内蔵AI）**
   - **仕組み**: 要約するコストを嫌い、直近N回のやり取りや、現在開いているファイルのみをコンテキストに含め、古い会話はFIFO（先入れ先出し）で純粋にコンテキストから「完全に消去」します。

### 自然言語による圧縮は本当に可能なのか？

結論から言えば、**「大枠の現状維持（非可逆圧縮）」としては有能ですが、「精密なルールの保持」としては限界があります。**

AI自身に「重要ではない不要な情報（ツールの失敗ログ等）を削って要約しろ」と指示した場合、以下のような現象が起きます。

- **残るもの**: 「現在〇〇の機能を実装中である」「DBへの接続エラーが未解決である」といった、**タスクの大きな方向性と直近のステータス**は綺麗に持ち越されます。
- **消失するもの（Lossy Compressionの弊害）**: 
  要約の過程で、AIは情報の抽象度を上げてしまいます。その結果、セッションの途中でユーザーが一度だけポロっと指示した**「個別の細かいコーディング制約（例: 日付フォーマットは常に YYYY-MM-DD とする、ここは let ではなく必ず const を使う等）」は、要約のノイズ判定に巻き込まれて高確率で欠落**します。

要約（自然言語圧縮）を経たエージェントは、「何を作るべきか」は覚えていますが、「どういうルールで作るかの細部」は忘れた状態になるのです。

## 3. モデルとエージェントの相互作用（シナジーと悪循環）

モデル（LLM）の特性と、エージェント（CLI）のコンテキスト管理が噛み合わなくなると、以下のような**「破綻の悪循環」**に陥ります。

1. **歴史の蓄積**: エージェントでの対話が長引き、コンテキスト（ログ）が肥大化する。
2. **モデルへの送信**: エージェントが大量の履歴をそのままモデルに送信する（Gemini CLIなどに顕著）。
3. **Lost in the middle の発生**: モデルが、ログの「中間」にある重要な前提条件（最初に読み込んだ `CODING_RULES.md` など）を見落とす。論点が薄まる。
4. **指示違反の出力**: モデルがルール違反のコードを出力し、テストやLintでエラーが発生する。
5. **ノイズの増幅**: エージェントが「エラーが出た、直せ」という情報を**さらにコンテキストに追加**してモデルに送る。
6. **完全な破綻**: 履歴がさらに長くなり、S/N比が最悪の状態になり、モデルが全く見当違いな修正をループし続ける。

逆に、Claude Codeのように自動で「圧縮（要約）」を行うエージェントの場合、(3)のLost in the middleは回避できるものの、圧縮時に(2)の「自然言語圧縮の限界」にぶつかり、**「いつの間にかコーディング規約を完全に忘れた状態でクリーンなコードを書き始める」**といった現象が起きます。

## 4. 対策：物理ファイルを経由した「セッションハンドオフ（State-to-Disk）」の徹底

エージェントの自動要約（Compaction）に頼ったり、無限の履歴を保持させたりするのではなく、**「確実な情報を物理ファイルに一旦書き出し、会話履歴を完全にリセットしてから、そのファイルだけを再び読み込む」**アプローチが最も有効かつ安全です。これは「State-to-Disk」とも呼ばれます。

「これまでの履歴は無視し、この要約ファイルとフォルダ構成だけを読んで再開して」という指示は、各エージェントの**機能（コマンドやUI）と組み合わせることで確実に機能します。**

### 各エージェントでの機能的な実現方法

1. **Claude (Claude Code)**
   - 対話の中で「現状と次やること」を `ai-stash/` 配下などのMarkdownに保存させます。
   - その後、**`/clear` コマンド**を実行します。これにより肥大化したチャットのメモリ（ノイズ）が完全に消去されます。
   - まっさらな状態で Claude は常に `CLAUDE.md` を自動ロードするため、プロジェクトの核となるルールは復元され、さらに先ほど書き出した Markdown のみを読み込ませることで、極めてS/N比の高いクリーンなセッションを再開できます。
2. **Gemini CLI**
   - 長大な履歴に引っ張られてエラーをループし始めた場合、`clear` 系の操作やプロセスを再起動して履歴を断ち切ります。
   - 起動時の引数や初期プロンプトで、対象のフォルダや作成済みの要約ファイルだけを明示的にパース（参照）させることで、無限のコンテキストに頼らない再スタートが可能です。
3. **Cursor / Codex系（エディタ内蔵型）**
   - **新しいチャット（`Cmd+L` で `+` アイコン）** または **「Cursor: Clear Context」**コマンドで、現在のスレッドのメモリを破棄します。
   - 新しい入力欄で **`@Folders`**（特定のディレクトリ）や **`@Files`**（書き出した要約ファイル）を明示的に指定することで、「システムプロンプト ＋ 指定した要約ファイルとフォルダ構成」**だけ**をコンテキストに乗せた高精度な推論を一瞬で実現できます。

### 運用上のベストプラクティスまとめ

- **重要な制約は常に「両端」に置く**: エージェントに明示的に指示を出す際、「絶対守るべきルール」はプロンプトの最後に添えるか、常にロードされる `CLAUDE.md` や `.cursorrules` 等に記述する（Primacy / Recency effect の活用）。
- **暗黙の要約を過信しない**: ツール付属の自動要約機能を信用せず、フェーズの切れ目では自分の目で確認できる「ドキュメント（物理ファイル）」として要約を出力させる。
- **能動的なリフレッシュ**: 悪循環（エラーループ等）の兆候が見えたら、エージェントが自力で直すのを待たずに、即座に履歴をクリア（`/clear`, 新規チャット）して仕切り直す。
