# AIエージェントにおけるコンテキストエントロピー

本ドキュメントでは、AIエージェントによる開発支援時の「コンテキストエントロピー」という概念と、モデル・エージェントの違い、および主要なエージェント（`gemini cli`、`claude`、`codex`）における特性の違いについて解説します。

## モデルとエージェントの違い

AIツールを利用する際、しばしば「モデル」と「エージェント」という用語が混同されますが、コンテキストの管理・扱いの観点からは明確な違いがあります。

- **モデル (LLM)**
  モデルそのもの（例：Gemini 1.5 Pro, Claude 3.5 Sonnet, GPT-4o など）が持つ「コンテキストウィンドウ」の物理的・アーキテクチャ上のサイズ限界のことです。
  例えば「コンテキストサイズが 1M（100万トークン）」と言った場合、それは**モデルが一度の推論で処理できる最大情報量**を指します。

- **エージェント (AIコーディングツール / CLI)**
  ユーザーとモデルの間に立って**対話やタスクを制御するソフトウェア**のことです。エージェントは、セッションの裏側でモデルへ送信するプロンプトを構築しています。
  モデルのコンテキストサイズが 1M であっても、エージェントは**セッション中のどの会話履歴をモデルに送るか、どのタイミングで情報を要約・圧縮・破棄するか**を独自に判断（または仕様として実装）しています。
  「モデルのコンテキストウィンドウが広いからといって、エージェントが過去の会話履歴をそのまま全部保持し続けているとは限らない」という点に留意する必要があります。

## コンテキストエントロピーとは

エージェントとの対話が長引くにつれて、プロンプト、返答、ファイルの読み込み結果、エラーログなど、多様な情報がエージェントのコンテキスト内（保持しているセッション履歴）に蓄積されます。
この情報の無秩序度（乱雑さ）が増大する現象を**コンテキストエントロピー**と呼びます。

### エントロピー増大による問題
- **焦点の喪失**: 過去に検討して破棄した実装案や不要なログがコンテキスト内に残り続け、エージェントが「今何を取り組むべきか」「どの制約を最優先すべきか」を見失いやすくなります（ハルシネーションの増加）。
- **S/N比（シグナル/ノイズ比）の低下**: 本当に必要な情報（シグナル）が、一時的なタスクから生じた大量の情報や試行錯誤の過程（ノイズ）に埋もれてしまいます。エージェントのパフォーマンス低下の主要因となります。

## 各エージェントのエントロピー管理と振る舞いの違い

各エージェントは、増大するコンテキストエントロピー（蓄積される会話履歴）に対してそれぞれ異なるアプローチで「要約・圧縮・切り捨て」を行います。

### 1. Claude (Claude Code / Claude CLI)
- **特徴**: トルク（会話の持続）をある程度保ちつつも、履歴のリミットに近づくか一定の条件を満たすと、エージェント側で**既存の会話履歴の自動要約（Summarization）**を行い、コンテキストを圧縮してからモデルに渡す傾向にあります。
- **注意点**: 圧縮機能により長いセッションを維持できますが、セッションの途中で提示した細かいルール（プロンプト）やコードのディティールが「要約によって抜け落ちる」リスクがあります。重要なルールは適宜リマインドするか、明示的にドキュメントから再度読み込ませる必要があります。

### 2. Gemini CLI
- **特徴**: 「Gemini 1.5 Pro」などの超大容量コンテキスト（1M〜2Mトークン等）と高精度な検索能力の強みを活かし、エントロピーが高まっても**履歴をそのまま保持し続け、必要に応じて参照する**力に長けていることが多いです。圧縮せずに生の状態で維持できる許容量が非常に大きいです。
- **注意点**: 大容量であるがゆえに「無駄な情報やノイズもすべて長期間保持」してしまいます。エントロピーが極端に高まると、過去の修正前コードやボツ案に意図せず引きずられることがあります。ユーザー側で意図的に「これまでの経緯は一旦忘れ、○○の要件のみに集中して」と指示（ソフトリセット）したり、セッションの切り替えを能動的に行う必要があります。

### 3. Codex (Codex系CLI / エディタ内蔵AI)
- **特徴**: ローカルエディタ上で高速かつ機敏なコード補完や修正を行うことを目的としているため、直近数十回のターン情報や、現在開いているファイル・手動で指定したファイルに強い重み付けを行います。古い履歴は「スライディングウィンドウ」的に**古いものから順に落としていく（忘れていく）**アプローチを取るケースが散見されます。
- **注意点**: 古い会話・指示が単純に押し出される形で消滅するため、セッションの初期に入力した前提条件が自然と無視されるようになります。重要な定義・ルールは、「常にコンテキストに入れるファイル（Pinned context）」として固定するか、ファイル（`doc/`や`ai-stash/`）の形で明記して常に参照させる構成が必要です。

## エージェントとモデルの相互作用（シナジーと悪循環）

エージェントがコンテキストをどう管理するか（圧縮するか、そのまま送るか）と、モデルがその情報をどう処理するか（一部を見落とすか、指示に従えなくなるか）の間には強い相互作用があります。
コンテキストエントロピーが増大すると、エージェント側の「適切な履歴管理の限界」とモデル側の「情報処理の限界（Lost in the middle等）」が同時に限界を迎え、**破綻の悪循環**に陥ることがあります。

## エントロピーへの対策（ベストプラクティス）

利用するエージェントに関わらず、コンテキスト内の S/N比 を高く保ち、エントロピーの増大を防ぐためには以下のような運用が不可欠です。

- **情報の外部化（ファイルへの退避）**: 議論の過程（How/Why）から得られた「結局どうするのか（What/So What）」という確定事項は、エージェントに頼らず即座に `ai-stash` や設計ドキュメント等の**ファイルに書き出します**。
- **能動的なセッション管理**: タスク（フェーズ）が切り替わるタイミングで、エージェントに「ここまでの成果と次の前提」を `ai-stash` に書き出させた後、**現在のセッションを破棄し、新しいセッションを立ち上げ直す**（コンテキストのクリア）運用を行います。
