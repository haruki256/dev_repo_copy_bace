# パッケージ配置ルール

## プロジェクト構成

本プロジェクトは、`doc/guide/STAGE_GUIDE.md` に基づいた構成を取ります。
基本構成（Stage 1/2）では単一の `src/` ディレクトリを使用します。
フロントエンド・バックエンド分離構成（Stage 3）を採用している場合は、それぞれのルートディレクトリ内でこのルールを適用します。

| ディレクトリ | 役割 | 言語 |
|---|---|---|
| `src/` | アプリケーションのメインソースコード | [Programming Language] |
| `doc/` | 共通ドキュメント | — |

## 汎用パッケージ構成 (`src/` 配下)

### app/（エンドポイント・画面単位）
- 1 つの入力インターフェイス（1 API、1 ジョブ、1 画面など） = 1 パッケージ
- パッケージ内に controller, service, data 等を配置
- controller (handler) は入力の受付と戻り値の返却（ルーティング/イベントフック）のみを担当
- ビジネスロジックは service に書く
- data にはそのパッケージ固有の構造体・型定義（Request/Response 等）を置く

### app/ の「モジュールグループ」の定義
- app/ 配下の単位は「UI画面」に限らず「ユーザー/外部からの入口（entrypoint）単位」で切る
- 画面単位以外の場合の目安：
  - バックエンドAPIの場合：エンドポイント単位（userApi/, orderApi/等）
  - CLIの場合：コマンド単位（importCmd/, exportCmd/等）
  - バッチの場合：ジョブ単位（dailyReport/, dataSync/等）
- 本ドキュメントでは便宜上「エンドポイント・画面」と併記するが、上記すべてを含む

### common/（技術的共通処理）
- utils/：汎用ユーティリティ（文字列処理、日時計算等）
- **commonに置けるもの：** 副作用なしの技術的な薄いユーティリティのみ
- **commonに置かないもの：** I/O（DB、HTTP、ファイル等）や状態を持つ処理

### shared/（共有リソース）
- dto/：複数モジュール（3つ以上）から参照されるデータ構造（ロジックは持たない）
- persistence/entity/：DB テーブルと 1:1 のデータ構造（DBを使用する場合）
- persistence/repository/：DB アクセス処理（DBを使用する場合）
- api (infra)/：外部APIやI/Oとの通信処理（外部サービスを使用する場合）

## モジュール内 data と shared/dto/ の使い分け

| | data（app/[Module]/data等） | shared/dto/ |
|---|---|---|
| 目的 | そのモジュール内で使う構造体 | 複数モジュールで共有する構造体 |
| 中身 | Request, Response, 中間データ | 横断で使う DTO |
| ロジック | 持たない | 持たない |
| 判断 | 1 グループでしか使わないもの | 3 グループ以上から参照されるもの |

## DB Entity（persistence/entity/）と DTO（shared/dto/）の区別

| | DB Entity（shared/persistence/entity/） | DTO（shared/dto/） |
|---|---|---|
| 目的 | テーブルの行を表す | 必要な項目だけ抽出・整形した受け渡し用データ |
| 内容 | カラムと 1:1 対応 | ビジネス上必要な項目のみ |
| ロジック | 持たない | 持たない |
| 管理 | テーブルごとに 1 つ（増殖させない） | 用途に応じて作成 |
| 配置 | shared/persistence/entity/ に集約 | shared/dto/ |

### なぜ DB Entity を集約するか (DB使用時のみ)
同じテーブルを複数モジュールから参照する場合、モジュールごとに DB 用データ構造を作ると増殖して「どれが正？」になる。
shared/persistence/entity/ に 1 テーブル 1 Entity で集約し、各モジュール側は必要な項目だけ data や dto に変換して返す。

### 注意：persistence/entity/ はDDD Entityではない
- persistence/entity/ はDBテーブルと1対1のデータ構造（ロジックを持たない。ORMモデルなど）
- DDD Entityは「振る舞い（ロジック）を持つオブジェクト」であり、別物

## shared/ に追加する際の判断基準
1. 3 モジュール以上から同じデータ構造を参照する → shared/dto/ に追加
2. データ変換が 2 箇所以上で必要 → shared/dto/ にマッピング用 DTO を追加
3. 上記以外のビジネスロジック → パッケージ内の service に書く（共通化しない）
4. 判断に迷ったらパッケージ内に置く

## 共通化の判断
- 「同じコードが 3 箇所以上」かつ「今後も増える見込み」→ 初めて共通化を検討
- 共通化する場合は DESIGN.md に影響範囲を明記すること
- 早すぎる共通化はしない

## 依存の方向ルール
- app/* → shared/* ：参照してよい
- app/* → common/* ：参照してよい
- shared/* → app/* ：**参照しない**
- common/* → app/* ：**参照しない**
- common/* → shared/* ：**参照しない**
- (DB使用時) persistence/entity/ を API 戻り値や画面描画パラメータにそのまま露出しない。app 側で data や dto に変換して扱う

## パッケージ内が複雑な場合の分割ルール
- 1 グループ内に明確に異なるユースケースがある場合、service を service/ フォルダ化し、ユースケースごとに分割する
- パッケージの単位は変えない
- 分割の目安：service 内に明確に異なる操作が 2 つ以上ある場合

## DDD要素について
- 本構成では簡易DDDとして「ユビキタス言語（GLOSSARY.md）」と「境界の判断基準（本ファイル）」のみ導入
- shared/dto/ はDDD Entityではない（ロジックを持たない受け渡し用データ構造）
- shared/persistence/entity/ もDDD Entityではない（DBテーブルと1:1のデータ構造）
- フルDDDの導入が必要になった場合は STAGE_GUIDE.md の Stage 3 を参照
