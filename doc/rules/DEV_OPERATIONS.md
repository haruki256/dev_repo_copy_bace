# 開発運用ルール

## このファイルについて
AI・開発者が開発作業中に守るべき運用ルールをまとめる。
コーディング規約は CODING_RULES.md を参照。

---

## AI が開発時に必ず実行すること

### 1. ドキュメントの読み込み順序
1. `AI_CONTEXT.md`（必読ルート）
2. 参照ファイル一覧に記載されたドキュメント（タスクに関連するもの）
3. `GLOSSARY.md`（ユビキタス言語の確認）
4. `CODING_RULES.md`（コーディング規約確認）

### 2. GLOSSARY.md の維持
- コード・ドキュメント内で新しいユビキタス言語が発生したら `doc/rules/GLOSSARY.md` に追記する
- 既存のユビキタス言語の意味が変わった場合は更新する
- 用語集の内容を他のファイルに複製しない（GLOSSARY.md へのパスを記載する）

### 3. `::UBIQ` タグの維持
- 新規ファイル作成時: ファイルの関与するドメイン概念を `::UBIQ` タグとしてファイル先頭に付与する（CODING_RULES.md 参照）
- ファイルの責務・影響範囲が変わった時: タグの用語を最新状態に更新する

### 4. ドキュメント目次（doc/README.md）の維持
- `doc/` 配下に新しいドキュメントを追加した場合: `doc/README.md` の適切なカテゴリにリンクと説明を追記する
- ドキュメントを削除・移動した場合: `doc/README.md` から削除・更新する

### 5. PACKAGE_STRUCTURE.md の維持
- ソースコード側に新しいファイル/フォルダを追加した場合: `doc/design/PACKAGE_STRUCTURE.md` のツリーに追記する
- ファイル/フォルダを削除・移動した場合: ツリーから削除・更新する

### 6. ARCHITECTURE_GUIDE.md の遵守
- パッケージ構成は入力・画面単位で切ること（`doc/guide/ARCHITECTURE_GUIDE.md` 参照）
- 新機能追加時は既存のパッケージパターンに従う
- パターンから逸脱する場合はユーザーに確認する

### 7. コード品質チェック
- コード修正後は Lint & Test を実行すること
- 例:
  ```bash
  [Test Command]
  ```
- 全てパスしてから次の作業に進む

---

## 修正時の影響範囲確認フロー

1. **`::UBIQ` タグで grep** — 修正対象のユビキタス言語に影響を受ける全ファイルを特定
   ```bash
   grep -r "::UBIQ.*[Term]" src/
   ```
2. **依存方向を確認** — PACKAGE_RULES.md の依存ルールに違反しないか確認
3. **GLOSSARY.md の用語を使用** — コミットメッセージ・コメントでも統一用語を使う
4. **テスト実行** — 修正範囲に関連するテストを実行

---

## 禁止事項

- GLOSSARY.md の用語を他のファイルに独自定義しない（参照パスを書く）
- `::UBIQ` タグなしで新規ファイルを作成しない
- 依存方向ルール（app → shared → common）に違反するインポートをしない
- `PACKAGE_STRUCTURE.md` や `doc/README.md` を更新せずにファイル構成を変更しない
